1. Context of testing
1.1 Project/test levels/test types:
Project Identification: This test plan identifies the testing strategies for two distinct target
applications (Systems Under Test):
• SUT 1 (OrangeHRM): A web-based HRM system used for Black-Box and Automation testing.
• SUT 2 (Library-DB-Express): A backend Node.js application used for White-Box testing.
Test Levels:
• Unit Testing: This level will be performed on SUT 2. The goal is to verify individual backend
functions and API endpoints in isolation to ensure code-level correctness.
• System Testing: This level will be performed on SUT 1. The application will be tested as a
complete, integrated system to validate end-to-end business requirements through the user
interface.
Test Types:
• Black-Box Testing (Functional): Focuses on testing the requirements of OrangeHRM without
internal code knowledge. The following techniques will be applied:
- Equivalence Class Partitioning (ECP) & Boundary Value Analysis (BVA): For
validating input fields and data constraints in the PIM, and Leave modules.
- Use Case Testing: To verify the "Add Employee" workflow within the PIM module.
- Decision Tables: To test the business logic of the "Leave Request/Approval" process in the
Leave module.
• Performance, Load, Stress, and Endurance (Soak) Testing: Performed on SUT 1 to evaluate
responsiveness, stability under peak loads, and long-term resilience under continuous traffic. This
ensures the application can handle expected traffic and sustained sessions without performance
degradation or connection exhaustion
• Compatibility Testing: Performed on SUT 1 to ensure the application’s core functionalities
(Login, PIM) remain consistent across different web browsers (e.g., Chrome, Firefox, Edge).• Usability Testing: A heuristic-based evaluation of SUT 1 to assess user-friendliness, navigation
efficiency, and the clarity of system feedback during key workflows.
• Automated Testing: Selenium will be used to automate repetitive functional test cases. Most
tests will be designed using Selenium IDE, while at least two key scenarios (e.g., Login and Add
Employee) will be exported and executed as scripts using Selenium WebDriver to demonstrate
scripting proficiency.
• White-Box Testing (Structural): Focuses on the internal logic and structure of the SUT 2
source code. It includes:
- Unit Testing: API endpoints and backend functions will be tested using Jest and Supertest
to ensure HTTP request/response cycles are handled correctly.
- Security Testing: Ensuring basic authentication mechanisms and authorized access to data
by verifying API endpoint protection, input validation, error handling, and resource access control
through URL parameters.
- Coverage & Mutation Analysis: Measuring test thoroughness through statement coverage
and using the Stryker tool for mutation testing to evaluate test suite robustness.

- White-Box Logic: Detailed verification of internal code paths and decision branches.

• Full-Scale Penetration Testing: While security audits and vulnerability assessments (such as

or external cyber-attack simulations are excluded.
• Visual/Aesthetic Testing: Purely cosmetic UI elements (specific color hex codes, exact font
sizes) are excluded, while functional visibility and layout consistency remain in-scope for
Usability and Compatibility analysis.

3. Test strategy
3.1 General:
The testing strategy is designed to validate both functional requirements and structural integrity. A
Risk-Based Testing approach is adopted for SUT 1 (OrangeHRM), focusing on high-priority
business workflows such as user management and leave processing. For SUT 2 (Library-DB-
Express), a Structural Testing approach is used to ensure code robustness through automated unit
testing and mutation analysis.3.2 Test levels:
• Unit Testing: Performed on SUT 2 to verify individual backend functions, API route controllers,
and database logic in isolation.
• System Testing: Performed on SUT 1 to validate the end-to-end functionality of the integrated
application through the user interface (GUI).
3.3 Test types:
• Black-Box Testing (Functional): Focused on validating SUT 1 requirements without looking at
the internal code, using techniques like ECP, BVA, and Decision Tables.
• White-Box Testing (Structural): Focused on the internal paths and logic of SUT 2, including
branch coverage and mutation testing.
• Automated Testing: Utilizing Selenium for GUI regression and Jest/Supertest for backend unit
verification.
• Non-Functional Testing: Includes Performance, Load, Stress, and Endurance (Soak) testing for
SUT 1 to ensure the system remains stable, responsive, and resilient during both peak traffic and
extended periods of concurrent user activity.
• Design & Compatibility Testing: Includes Usability and Compatibility testing for SUT 1 to
verify cross-browser stability and user-friendly navigation.
• Security Testing (SUT 2): Focused on identifying vulnerabilities within the backend logic, such
as information disclosure via stack traces, resource access control, and endpoint resilience against
malformed inputs.

• Usability Evaluation Report: A document detailing the findings of heuristic evaluations and UX

• Compatibility Matrix: A cross-browser testing report showing functional consistency across

• Security Audit & Vulnerability Report: A technical document detailing SUT 2 backend

• Coverage & Mutation Reports: Quantitative analysis reports showing:
- Statement and Branch coverage percentages.
- Stryker Mutation Analysis results to verify test suite robustness.
• Test Completion Report (Final Test Summary Report): A comprehensive final report
summarizing the testing activities, pass/fail ratios, and final quality assessment of both SUT 1 and
SUT 2.
3.5 Test design techniques:
• For SUT 1:
- Functional: Equivalence Partitioning (EP), Boundary Value Analysis (BVA), Use Case
Testing, and Decision Tables.
- Non-Functional & UX: Heuristic Evaluation (for Usability), Cross-Browser Verification (for
Compatibility), and Quantitative Performance Modeling (for Load, Stress, and Endurance
testing).
• For SUT 2: Statement and Branch Coverage (via Jest/Istanbul), Mutation Testing (via Stryker),
and Logic-based Security Verification. Security testing specifically focuses on:
- Input Validation: Ensuring resilience against XSS and SQL Injection.
- Error Handling: Verifying that the system prevents information disclosure (e.g., stack trace
leaks).- Resource Access Control: Testing endpoint protection against unauthorized URL parameter
manipulation.
3.6 Entry and exit criteria:
Entry Criteria: System test execution shall commence only when the following conditions are
met:
• Approval: The Master Test Plan is finalized and signed off by the project team.
• Documentation: All test cases (Black-Box and White-Box) are documented and approved.
• Environment Readiness: The SUT 1 (OrangeHRM) demo environment is accessible, and the
SUT 2 (Library-DB-Express) local environment is correctly configured with Node.js and SQLite.
• Test Data: Synthetic datasets for ECP, BVA, and security payloads (XSS/SQLi) are prepared and
ready for injection.
Exit Criteria: Testing activities are considered complete when:
• Functional Completion: 100% of high-priority test cases for SUT 1 have been executed, with all
"Critical" and "High" severity defects resolved.
• Structural Integrity (SUT 2): A minimum of 95% Statement Coverage and 90% Branch
Coverage is achieved (Reflecting the high quality verified in the White-Box Report).
• Robustness Verification: The mutation score must meet the defined threshold, confirming that
the test suite is capable of detecting logic faults.
• Security & Risk Assessment: All identified security vulnerabilities (specifically Stack Trace
leaks and DoS risks) are documented with a final impact analysis.
• Defect Management: Any remaining Severity 2 or 3 defects are reviewed and logged in the Final
Test Summary Report for future maintenance.
3.7 Test completion criteria:
Testing is finalized when the Exit Criteria are met and the Final Test Summary Report is signed off
for presentation.
3.8 Metrics to be collected:
• Test Case Execution Rate: The percentage of planned test cases that have been executed.
• Test Case Pass Rate: The ratio of passed tests to total tests executed, tracking the overall
functional reliability.• Defect Severity Distribution: The classification of identified issues by impact (Critical, High,
Medium, Low) to prioritize risk management.
• Code Coverage (SUT 2): Quantitative measurement of Statement and Branch coverage via Jest/
Istanbul to ensure structural thoroughness.
• Mutation Score (SUT 2): Evaluation of the test suite’s robustness and fault-detection capability
using Stryker.
• Performance Benchmarks (SUT 1): Measurement of Average Response Time, Throughput, and
Error Rates to evaluate system stability under different load conditions.
• System Breaking Point: Identification of the load level where system performance degrades
beyond acceptable reliability thresholds.

This section specifies the origin, characteristics, and management of the data required for both
SUTs:

and Boundary Value Analysis (BVA) tables.
- SUT 2: Data is sourced from the backend project repository provided by the lecturer.
• Specific Data Requirements (SUT 1):

validity, and balance sufficiency as defined in the Decision Table.

sensitive personal data is required; all data is synthetic/mock.
• Responsibility: The Test Analyst and SDET Specialist are responsible for preparing and
maintaining the test datasets.
